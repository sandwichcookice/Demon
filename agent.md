### 目的

本文件旨在對目前的 Demon 專案進行全面的測試與驗證，確保所有核心功能模組、插件管理、對話流程控制、歷史紀錄管理等均符合預期要求，並能在最終發佈版本（Version 1.0）前達到穩定可靠的狀態。以下將針對各個模組進行詳細測試，並確保其功能完整性與錯誤處理機制健全。
對目前已經存在的腳本進行最後審查，並將所有的測試檔案都重構，以便他們能夠適配下面的要求。

## 插件管理與優先載入

確認 `PluginsManager` 能正確管理所有插件的生命週期，包括上線 (`online`)、下線 (`offline`)、重啟 (`restart`) 等操作。檢查插件啟動順序是否依據各插件設定的 `priority` 值執行，確保相依的插件能按優先權正確初始化。在重複呼叫上線時，系統應能識別並避免插件重複啟動（`queueOnline` 重複上線檢查），並確保 `getPluginState` 等狀態查詢接口回傳正確的數值狀態。確認整體插件管理機制穩定，如發生異常情況（例如啟動失敗）亦有適當的錯誤日誌紀錄。

## 對話流程控制（TalkToDemon）

檢查核心對話管理模組 `TalkToDemon` 的流程：確保對話佇列機制運作正常，包含訊息的排隊、插隊及中斷處理。測試 `talk()` 對話輸入的各種情境：**(1)** 當前無對話進行時能立即處理新訊息；**(2)** 當有對話進行中時，新的可中斷訊息能正確中止現行串流並啟動處理新訊息；**(3)** 標記為不可中斷（`uninterruptible`）的訊息在忙碌時會被忽略，標記為重要（`important`）的訊息則進入待處理隊列。驗證對話歷史與工具結果會正確注入提示詞：`TalkToDemon` 應透過 `historyManager` 讀取先前對話記錄，並配合 `PromptComposer` 組合當前訊息與暫存的工具結果。確定串流輸出過程中的事件（`data`、`end`、`error`、`abort`）皆有適當處理，例如手動中止 (`manualAbort`) 時能即時停止 LLM 輸出並關閉當前任務。整體確保對話流程控制穩定，能按設計在多訊息情境下維持正確的對話狀態轉換。

## 歷史紀錄管理

確認對話歷史模組 (`historyManager`) 對使用者歷史訊息的持久化儲存與擷取功能正常。測試 `appendMessage` 能將每則對話訊息正確追加至對應使用者的歷史檔案（例如 `history/<使用者ID>.json`），並保留角色類型與內容。使用 `getHistory` 讀取歷史時，應依參數正確返回最近的 N 筆訊息，且超過限制時只取最新的片段（例如請求最近 3 筆時只返回最新的三則訊息）。檢查歷史記錄裁剪機制是否生效，例如超過設定的保存上限或時限（50則或10分鐘）時能淘汰最舊記錄。驗證在歷史檔案不存在或讀取失敗的情況下，模組會妥善處理（回傳空陣列並記錄警告而不會導致崩潰）。確保 `clearHistory` 等功能可正常清除指定使用者的歷史紀錄檔案。

## 提示詞組合與上下文注入（PromptComposer）

確認 `PromptComposer` 模組能將系統提示詞、歷史對話與當前使用者訊息正確組合，產生發送給 LLM 的訊息陣列。測試在無工具使用時，`composeMessages` 輸出僅包含對話歷史（經過裁剪）和使用者最新訊息；若先前有工具結果暫存於 `toolResultBuffer`，則應將這些結果適當地插入提示內容中，再交由 LLM 處理。驗證當輸入參數不符合預期格式時，`PromptComposer` 能捕捉例外不致中斷整個對話流程，並產生錯誤日誌提醒。另外確認系統是否有預設的系統提示詞或角色設定，如有則需確保這些內容在組合訊息時被包含並正確地置於對話開頭。總體而言，確保提示詞組合模組能穩定地提供 LLM 所需的完整上下文。

## 工具呼叫處理與輸出路由

檢查 LLM 工具呼叫機制，包含 `toolOutputRouter` 串流檢測與 JSON 解析、以及工具執行結果的處理流程。首先，驗證 `toolOutputRouter` 能即時監控 LLM 串流輸出內容，正確辨識出其中的工具呼叫 JSON 結構（帶有 `toolName` 欄位的 `{...}` 物件）。在檢測到完整的 JSON 時，路由器應中斷繼續向使用者輸出該段內容，並透過 `handleTool` 呼叫對應的插件功能。測試當工具執行過程超過預定逾時（預設 1500ms）時，`handleTool` 會正確觸發逾時錯誤並回傳失敗結果。確認執行插件成功時，返回的結果會封裝成對話訊息物件插入（`success: true` 並附加結果資料），失敗或逾時則回傳標記 `success: false` 的工具訊息。`toolOutputRouter` 在工具執行期間應發布 `waiting` 事件切換系統為等待狀態，確保使用者端收到「正在查詢，請稍候」的提示。最後，驗證工具執行結束後，系統會將結果訊息加入對話歷史重新組合提示詞，重新向 LLM 發送最終問答請求。整體確保 LLM 工具呼叫流程順暢，工具 JSON 被正確攔截執行，結果再正確地反映在最終回覆中。

## LLM 模型插件（LlamaServer）

針對內建的 LLM 插件（`llamaServer`）進行全面檢查，包括本地、遠端、伺服器三種模式的啟動與切換。**本地模式**：確認 `llamaServerManager` 能在本地正確啟動推理服務，例如載入模型並開啟本地 API（預期在 localhost:8011 提供類似 OpenAI 接口）。透過 `online` 方法啟動本地 LLM 時，應傳回成功訊息或物件，`state()` 回傳 1 表示服務運行中。測試串流回應機制：呼叫 `send(messages)` 應返回一個 `EventEmitter` 串流物件，能持續收到部分回答內容並在結束時觸發 `end`。確認 `abort` 功能有效，當外部呼叫中止時，`EventEmitter.abort()` 能即時終止底層請求（銷毀 HTTP 流）並觸發 `abort` 事件。**遠端模式**：如果有整合第三方雲端模型服務，確認提供的 API 金鑰或 URL 在設定中正確配置，遠端模式下的 `online` 可以正常連接外部服務（例如 OpenAI API），`send` 能取得回應。不論本地或遠端，`updateStrategy('auto')` 自動模式應能依環境自動選擇策略（例如遠端可用時優先遠端）。**伺服器模式**：啟用時應同時啟動本地服務並註冊 ngrok 子網域，允許透過公開 URL 進行對話請求。測試對應的 HTTP 接口（POST `/llama/<action>`）是否按設計運作：傳入對話消息列表應透過 SSE（Server-Sent Events）串流回傳模型輸出。最後，驗證各模式切換 (`updateStrategy`) 時資源清理與重新初始化正常，避免遺留舊的執行緒或服務進程。確保 LLM 插件整體穩定，能在不同運行模式下提供預期的對話生成功能。

## Discord 介接插件

全面測試 Discord 插件，確保機器人能正常連接並響應各種互動。在 **連線** 方面，確認提供的 `config.js` 設定（Bot Token、應用 ID 等）正確載入，`online()` 能成功登入 Discord 平台，`offline()` 能登出且釋放連線資源。**訊息監聽**：啟動後，機器人應全域監聽所有頻道中的訊息事件。測試私訊（DM）、提及機器人 (@標註) 以及直接回覆機器人訊息三種情境：預期插件的 MessageHandler 會攔截這些訊息並將內容傳遞給 `TalkToDemon` 進行對話。驗證當非授權使用者發話時，機器人會回應預設的拒絕語（如：「我還學不會跟別人說話」），僅對特定授權對象的訊息提供回覆。**指令功能**：確認已註冊的 Slash 指令（例如 `/ping`）能正確觸發並回應預期結果（如回傳 “pong” 或相關訊息），在未指定 guild 的情況下指令應註冊為全域可用。**發送訊息**：透過測試 `send` 方法調用 Discord 插件內部功能，包含 `send`、`restart` 等，確保例如 `plugins.discord.send({ func: 'send', channelId: X, message: 'test' })` 能讓機器人在指定頻道發出對應訊息。同時驗證錯誤處理：例如提供無效的頻道ID或內容時，插件能適當紀錄錯誤而不崩潰。最後，檢視插件在運作過程中的日誌輸出與註解是否詳細，確保任何 Discord API 錯誤（如連線中斷、權限不足）都有對應的警告或錯誤日誌供追蹤調試。

## 語音辨識插件（ASR）

檢驗 **語音辨識（Automatic Speech Recognition）插件** 在不同策略下的功能表現。**本地模式**：確認本地 ASR 能啟動麥克風或語音引擎進行語音轉文字轉換，`online()` 執行後狀態變為上線，並能對提供的音訊進行及時轉錄（若有對應測試接口或模擬輸入）。測試在沒有外部輸入時不會產生例外錯誤，且 `state()` 回傳正確狀態碼。**遠端模式**：應用 ngrok 代理，由外部請求觸發 ASR。啟動遠端模式後（可能需要先啟動 ngrok 插件），確認 Demon 在指定子網域（例如 `/asr`）註冊了 HTTP 接口。模擬對公開 URL 發送相應指令（如 `POST /asr/start` 開始聆聽、`POST /asr/stop` 停止）時，伺服器應呼叫本地 ASR 對應的功能並返回成功狀態碼 200。**伺服器模式**：類似遠端模式，確認 Demon 在啟動 ASR 插件伺服器策略時，會自動將本地 ASR 上線並透過 ngrok 開放控制接口。測試 `restart` 能同時重啟本地 ASR 及重新註冊 HTTP 路由。檢查錯誤與例外處理：例如當本地 ASR 引擎或 Python 腳本發生錯誤時，插件能記錄日誌（`logger.error`）且 HTTP 請求得到適當的 500 錯誤響應。最後，確保 ASR 輸出的文字結果能正確地回傳或傳遞給系統其它部分（例如作為工具輸出給 TalkToDemon 或供後續 TTS 使用），整個語音辨識流程穩定可靠。

## 語音合成插件（TTS）

驗證 **文字轉語音（Text-to-Speech）插件** 的各項功能。**本地模式**：上線後應初始化語音引擎或透過 Python 腳本準備語音合成功能，`send(text)` 傳入文字時能在本地裝置播放或生成對應語音（如使用系統 TTS 引擎或外部程式）。檢查本地模式下連續多次呼叫 `send` 的行為，確保插件能順序播放或處理，不會發生競態條件。**伺服器模式**：啟用時會先上線本地 TTS，再透過 ngrok 將 TTS 功能公開為 HTTP 接口。確認 Demon 在對應子網域（如 `/tts`）註冊路由後，外部可透過 `POST /tts/send` 向其傳送要轉換的文字。測試該介面對不同輸入的反應：正常情況下傳入 `{"text": "hello"}` 應返回 200 並觸發本地語音播放，若請求格式錯誤或本地語音引擎失敗則返回 500。檢查 TTS 插件在 `offline()` 時能確實停止或重置語音引擎，並解除已註冊的 ngrok 子網域。**整合測試**：配合 SpeechBroker 插件驗證，在 Demon 串流輸出文字時，TTS 插件能接收到正確的文字並立即朗讀。特別地，當 TTS 插件未上線時，SpeechBroker 應能偵測到並記錄警告而不嘗試送出語音。最後，審視 TTS 插件的錯誤處理與日誌，例如語音引擎不存在或語音合成失敗時有無詳盡的錯誤紀錄，確保整個文字轉語音流程的穩定性和可追蹤性。

## 語音串流中介插件（SpeechBroker）

檢查 **SpeechBroker 插件** 將 LLM 文本串流轉接至 TTS 語音播報的功能。上線後，SpeechBroker 應註冊對 `TalkToDemon` 串流事件的監聽，即在每次 LLM 輸出文字片段時觸發自訂處理。驗證句子解析邏輯：SpeechBroker 應累積輸出片段至偵測到句尾標點（包含中英文字號如句點、問號、驚嘆號、波浪號等）時，將完整句子（含先前累積部分）經過清理後送交 TTS 插件發聲。檢視 `sanitizeChunk()` 功能是否按預期運作：需移除括號表示的表情文字並將全形標點轉為半形或去除特殊符號（如 ♥ 等）。測試在每個句尾，SpeechBroker 都呼叫一次 `sendToTTS` 並清空緩存，確保不會漏掉句尾部分。當串流輸出結束 (`end` 事件) 或被中止 (`abort` 事件) 時，如尚有殘留未播報文字，插件應自動在句末加上句點後將其傳送至 TTS，避免最後一段話丟失。確定當 TTS 插件未在線時，SpeechBroker 會記錄警告且跳過發送步驟，不會因無法呼叫 TTS 而拋出錯誤。最後，驗證 SpeechBroker 在 `offline()` 時能移除所有事件監聽器並重置狀態，以防止下線後仍繼續接收資料導致重複播報。整體確保 SpeechBroker 實現了即時逐句播報，沒有明顯延遲或漏播，並對異常情境具有適當的處理。

## 外網通道插件（ngrok）

測試 ngrok 插件用於開啟外網通道的整合功能。**啟動與關閉**：調用 `plugins.ngrok.online(options)` 後，應在後端啟動一個 ngrok 子程序並監聽指定埠（預設 3000），同時建立 Express 本地伺服器。確認 `online` 回傳的公開 URL 有效（如 `*.ngrok.io`），`state()` 在運行時回傳 1。測試當 ngrok 執行檔路徑錯誤或無法啟動時，系統能捕捉錯誤並記錄（例如 `無法啟動 ngrok: ...` 的日誌）。**子網域註冊**：利用 `plugins.ngrok.send(...)` 介面註冊 HTTP 路由，傳入參數 `{ action: 'register', subdomain: '<名稱>', handler: function(req,res) {...} }` 後，Express 應開始攔截對應 `<名稱>` 路徑的請求並執行給定的 handler。模擬發送 HTTP 請求到公開網址的 `<subdomain>` 路徑，驗證請求能到達註冊的處理函式並返回預期結果（例如我們設定 handler 回傳 JSON `{ok:true}`，則通過 ngrok URL 獲得相同響應）。測試重複註冊相同子網域時插件會拒絕並警告「子網域已被註冊」，註冊不存在的子網域或成功解除後再次解除時亦有相應警告。**解除註冊**：調用 `send({ action: 'unregister', subdomain: X })` 後，確認該路徑處理被移除，對應請求改回傳 404。最後，調用 `offline()` 應關閉本地 Express 伺服器並終止 ngrok 程序，`state()` 回傳 0。整體確保 ngrok 插件能穩定地建立/關閉外網通道，並可靠地註冊/撤銷子路由轉發插件請求，同時對錯誤情況有完善的處理和日誌。

## 工具清單插件（ToolReference）

確認 ToolReference 插件正確收集並提供系統中的工具描述清單。上線時（`online()`），插件應掃描 `plugins/` 目錄下所有插件的 `tool-description.json` 文件，將每個插件的工具說明載入快取。驗證在正常情況下，`send()` 呼叫會回傳一個包含各工具描述的物件，其中鍵為插件名稱、值為該插件對應的描述 JSON。確保描述內容符合預期格式，例如每個描述應至少包含工具名稱、功能說明以及輸入/輸出範例。若某插件缺少描述檔或 JSON 解析失敗，插件應跳過該項並記錄警告但不中斷整體收集流程。測試當 Demon 新增或移除插件後，重啟 ToolReference 插件能更新清單以反映最新狀態，不會遺留不存在的舊條目。最後，確認 ToolReference 的 `pluginType` 標識為 `LLM`且已在 `PluginsManager` 中註冊為特殊查詢插件，確保 LLM 模型可透過查詢 ToolReference 取得最新工具清單，用於後續對話中的工具選擇。整體確保工具清單資料完整且即時，提供的資訊可以支援 LLM 正確理解可用工具集合。

## 系統資訊工具（OsInfor）

驗證 **OsInfor 工具**提供的系統資訊查詢功能。調用 `OsInfor.table()` 預期取得包含作業系統主要資訊的物件，諸如平臺 (`platform`)、架構 (`arch`)、主機名稱 (`hostname`)、系統版本 (`release`/`type`) 等欄位。測試連續兩次呼叫 `table()` 時是否使用快取機制（第一次查詢後應緩存結果，再次調用直接返回而不重複計算），以確保效能。調用 `OsInfor.get('<field>')` 應能單獨擷取特定欄位值；驗證對有效欄位（如 `'platform'`）返回正確內容，對不存在的欄位名稱應有對應的處理（可能拋出錯誤或回傳 undefined）。檢查在查詢過程中如發生例外（例如底層 `os` 模組調用失敗），模組會記錄錯誤日誌並將錯誤向上拋出。此外，確認該工具未在正常情況下對終端使用者輸出敏感資訊，只提供開發或插件內部使用。確保 OsInfor 提供的系統資訊正確且可靠，可被其他插件或功能模組（如健康檢查或環境決策）引用。

## 錯誤處理與日誌紀錄

全面檢視整個系統的錯誤處理機制與日誌紀錄，確保各模組在異常情況下都有妥善處理而不影響服務穩定。首先，在所有插件和核心模組中引發的例外都應被捕獲並記錄：例如 Discord 連線錯誤、ngrok 啟動失敗、插件執行過程中的未預期錯誤等，都需要在日誌中留下錯誤訊息以供追蹤。確認 `Logger` 日誌工具運作正常：每個模組所產生的 `.log` 檔案（如 `TalkToDemon.log`、`speechBroker.log` 等）正確輸出資訊，`Original` 模式可讓日誌輸出原始物件內容，`UseConsole` 設定能控制是否同步輸出至主控台。測試各種錯誤情境下系統的反應，例如 historyManager 碰到檔案讀寫問題應僅警告而不中斷對話流程、工具執行逾時會回傳失敗結果但不影響後續對話進行。另外，驗證敏感資訊不會記錄在日誌中（如 Discord Token 等應避免明文出現在日誌檔）。最後，確保日誌檔案與錯誤處理流程文件化且易於理解，方便開發者在最終發行版（Version 1.0）前進行審查與除錯，使產品在面對各種異常時都具有可預期的表現。

## 設定檔與資料儲存

檢查系統中各項設定與資料的讀寫儲存機制，確保配置檔案正確加载且持久資料保存完好。**插件設定**：例如 Discord 插件的 `config.js` 應包含 Bot Token、頻道 ID 等必要設定，啟動時應正確讀取該檔案；測試當檔案缺漏或內容不完整時，插件能立即拋出錯誤避免使用預設空值繼續執行。確認其他可能的設定（如 OpenAI API Key、llama 模型路徑等）都已在適當位置配置，並在啟動對應功能時正確讀入。**資料保存**：對話歷史採用 JSON 檔案存儲於 `history/` 目錄下，每位使用者各自一檔，驗證這些檔案讀寫正常，並隨時間及大小正常滾動（例如未無限增長侵佔磁碟，舊紀錄依策略裁剪）。**檔案路徑**：檢查程式中使用的路徑皆相對正確，如 ngrok 執行檔預設路徑 `Server/ngrok/ngrok.exe` 是否存在，llama 本地服務的相關檔案路徑是否有效，語音插件所需的資源（如 Python 腳本或模型檔）是否放置於專案預期位置。**備份與預設**：如果專案附帶 `example` 或範本設定檔（例如 Discord `config.example.js`），確認最終版本已將其移除或明確標示需要用戶提供實際值。此外，驗證系統在無預先配置某些非必要檔案時能平穩啟動（跳過相關功能或給出明確錯誤），在有所有必要設定時能正常執行全部功能。確保版本 1.0 發佈時，所有設定與資料儲存項目都已齊全，且讀寫機制健全可靠。
